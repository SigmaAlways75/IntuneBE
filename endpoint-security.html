Security Stack Overview

Intune's security stack combines compliance evaluation, attack surface reduction, encryption enforcement, and real-time threat signals from Defender for Endpoint.

✅
Compliance Policies
Define the minimum security bar — OS version, encryption, PIN, jailbreak. Device gets a Compliant/Non-Compliant state used by Conditional Access to grant or deny resource access.
CA uses compliance state
🛡️
Endpoint Security Policies
Attack Surface Reduction rules, antivirus settings, disk encryption (BitLocker/FileVault), Windows Firewall rules, and account protection — all pushed via Intune MDM.
Policy priority order
🔍
Microsoft Defender for Endpoint
MDE risk score feeds into Intune compliance evaluation. If a device gets malware, it instantly becomes non-compliant and is blocked from all corporate resources.
MDE integration details
🔐
Encryption Enforcement
BitLocker (Windows), FileVault (macOS), and device storage encryption enforced via compliance. Recovery keys escrowed to Intune/Azure AD for IT-managed recovery.
🎯
Attack Surface Reduction
ASR rules block common attack vectors: Office macros, credential theft from LSASS, executable content from email, untrusted USB devices. Pushed via Intune to all enrolled Windows endpoints.
👮
Security Baselines
Microsoft-curated hardening templates for Windows, Edge, and Defender. Pre-configured with CIS/NIST-aligned settings. One-click deploy to bring all endpoints to a hardened baseline.

Non-Compliance Actions
⏱️
Grace Period
Configure a grace period (e.g. 3 days). Device shows as non-compliant but isn't blocked yet. User gets notification to remediate. After grace period, blocking begins.
📧
Email Notification
Intune automatically emails the user and/or IT admin when a device becomes non-compliant. Custom email templates with branding supported.
🔒
Resource Block via CA
Conditional Access checks compliance flag. Non-compliant device token is denied. User sees "You can't access this right now" in Outlook, Teams, SharePoint.
Conditional Access scenarios
🗑️
Retire / Wipe
As a final action after extended non-compliance, IT can trigger a remote retire (remove corporate data) or full wipe automatically.
MDE Integration
Microsoft Defender for Endpoint Signal Flow

MDE provides real-time threat signals that Intune consumes to dynamically update device compliance — creating a live security feedback loop.

🔄 How the Signal Loop Works
1
MDE Sensor Detects Threat
MDE sensor (built into Windows 10/11) detects suspicious activity, malware, or active attack on the endpoint.
2
Risk Score Assigned
MDE assigns a Machine Risk Score: Clear / Low / Medium / High based on the severity and confidence of the detection.
3
Intune Compliance Evaluated
Intune compliance policy checks: "Is MDE risk score ≤ configured threshold?" If MDE flags High risk and policy requires Medium or below → non-compliant.
4
CA Blocks Access
Conditional Access sees non-compliant flag → blocks all M365 access. User blocked until MDE clears the threat and risk score drops.
CA token flow
🔴
Critical — Ransomware
MDE detects file encryption activity. Risk → High. Intune marks non-compliant. CA blocks all M365 access within minutes. IT alerted via Defender portal.
🟠
High — Credential Harvesting
LSASS memory dump detected. ASR rule fires. MDE risk → High. Device quarantined from corporate network via CA block.
🟡
Medium — Suspicious PowerShell
Obfuscated PowerShell executed. MDE risk → Medium. If policy requires Low or below, device becomes non-compliant. User gets email notification.
🟢
Low — PUA Detected
Potentially Unwanted Application found. MDE risk → Low. Most policies allow Low risk. Alert sent but device stays compliant. IT can investigate from Defender portal.

Zero Trust
Intune's Role in Zero Trust Architecture

Zero Trust: "Never Trust, Always Verify." Intune contributes the Device Health Signal — one of the core pillars in the Microsoft Zero Trust model.

🔍 Device Signals Intune Contributes
→ Is the device enrolled and managed? (Intune)
→ Is the device compliant with security policy? (Intune Compliance)
→ Does the device have active threats? (MDE risk score)
→ Is the device using a trusted platform? (TPM attestation)
→ Is the user's identity verified? (Entra ID MFA)

All signals combined → Conditional Access makes the access decision.
Conditional Access policy engine
🔑
Identity Pillar
Entra ID handles identity. MFA, passwordless (FIDO2/WHfB), Identity Protection risk signals. Intune ensures device identity is registered and compliant.
💻
Device Pillar
Intune owns this pillar. Enrollment, compliance, configuration enforcement, health attestation. Device health certificate from TPM proves boot integrity.
📦
Application Pillar
App Protection Policies (MAM) protect data inside apps. Defender for Cloud Apps monitors SaaS usage and blocks risky app behaviours.
App protection policies
📊
Data Pillar
Microsoft Purview classifies and labels data. Intune MAM policies enforce data handling rules — no save to personal storage, no copy to personal apps.
🌐
Network Pillar
Per-app VPN and Always-On VPN pushed via Intune. Microsoft Tunnel provides secure access to internal resources from managed mobile devices.
Microsoft Tunnel & VPN
🔭
Visibility & Analytics
Microsoft Sentinel (SIEM), Defender XDR, and Intune reports. Endpoint Analytics tracks device health trends. Log Analytics for deep investigation.
Reporting via Graph API
