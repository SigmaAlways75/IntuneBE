📊 Analytics & Reporting

Intune Reporting — Current Issues & Microsoft's Roadmap

A frank look at the known limitations in Intune's current reporting and analytics capabilities, the root causes behind them, and the concrete roadmap items Microsoft has published to address each one.

CategoryReporting & Analytics
AffectsAll tenants
Horizon2025–2026 roadmap
01
Current State
Why Intune Reporting Falls Short

Intune's reporting has historically been fragmented across multiple locations — the legacy portal, the new Intune admin centre, Endpoint Analytics, and Log Analytics. This causes inconsistencies, stale data, and gaps that IT admins encounter daily.

⏱️
Reporting Latency
Compliance and inventory data in the Intune portal can lag 15–30 minutes behind actual device state. During incidents or rapid remediation workflows, admins are making decisions on stale snapshots rather than live state.
🗂️
Fragmented Report Locations
Reports are scattered: device compliance in one node, app install status in another, Endpoint Analytics in a separate blade, and raw logs only in Log Analytics. There is no single pane of glass.
📉
Limited Export Scale
The built-in export cap was 1,000 rows for several report types. Organisations with 50,000+ devices couldn't export full datasets without scripting Graph API calls manually.
🔄
Inconsistent Refresh Cadence
Different report types refresh on different schedules — some hourly, some every 4 hours, some only on-demand. This inconsistency makes it hard to build reliable operational dashboards.
🔍
No Cross-Platform Unified View
Compliance status for Windows, iOS, Android, and macOS lives in separate report tiles. There is no single unified query interface across all platforms without exporting to external tools.
📊
Endpoint Analytics Accuracy
Endpoint Analytics startup performance and app reliability scores are computed from sampled telemetry, not 100% of devices. Anomalies can be missed, especially in small or heterogeneous fleets.
⚠️
Real-World Impact
The most painful consequence of reporting gaps is the compliance audit gap: a device can be non-compliant for several minutes to hours before Conditional Access picks up the state change. During this window, a compromised or out-of-policy device retains access to corporate resources. Microsoft's Continuous Access Evaluation (CAE) reduces this window but does not eliminate it entirely. 02
Root Causes
Why These Issues Exist — Technical Background

These limitations are not random — they have specific architectural causes rooted in how Intune's reporting pipeline was originally designed.

The Snapshot vs Live Architecture Problem

Intune's core reporting pipeline was designed around periodic device check-ins (every 8 hours) rather than real-time event streaming. Compliance state is computed from the last check-in snapshot, not a live query to the device. This means reports are only as fresh as the last sync.

Data Warehouse Architecture

The Intune Data Warehouse was the original reporting backend — a read-only OData endpoint refreshed daily. It was designed for historical trend analysis, not operational dashboards. Many report types still pull from this warehouse rather than the more current Graph API sources.

The Graph API Difference
The Graph API /deviceManagement/managedDevices endpoint returns the current known state from Intune's operational database — not the data warehouse. For compliance monitoring scripts and dashboards that need the most current data, always use the Graph API directly rather than the portal's built-in reports. Graph API key endpoints

04
Microsoft Roadmap
How Microsoft Plans to Fix This

Microsoft has acknowledged the reporting gaps and has published a series of improvements across Intune, Endpoint Analytics, and the broader Microsoft 365 monitoring stack.

Published Roadmap Items (2025–2026)

🔄
Advanced Reporting Refresh (GA 2025)
Microsoft overhauled the report generation engine to increase refresh frequency for compliance and configuration reports. The new engine pulls from the Graph API operational database rather than the legacy data warehouse for most report types.
📊
Unified Reporting Node
Consolidation of all Intune reports under a single 'Reports' node in the admin centre — eliminating the scatter across Devices, Apps, and separate blades. Rollout ongoing across report categories.
📤
Unlimited Export via Graph
The 1,000-row export cap was removed for most reports. Large exports now use an asynchronous pattern: trigger report generation via Graph, poll for completion, then download the full file. Supports millions of rows.
🔗
Power BI Integration Improvements
Updated Power BI connector uses the Graph API data source instead of the legacy OData Data Warehouse. Faster refresh, more current data, and support for delta queries to reduce Power BI dataset refresh time.
⚡
Real-Time Compliance via CAE
Continuous Access Evaluation (CAE) enables near-real-time access revocation when compliance changes, without waiting for token expiry. Intune compliance state changes are propagated to Entra ID within minutes rather than hours.
🤖
Copilot for Security Integration
Microsoft 365 Copilot integration with Intune (in preview) allows natural language queries over device data — 'show me all devices with outdated OS that have access to SharePoint' — using Graph API under the hood.

Recommended Architecture for Better Reporting Today

1
Stream Diagnostics to Log Analytics
In the Intune admin centre, configure Diagnostic Settings to stream device compliance, configuration, audit, and operational logs to an Azure Log Analytics workspace. This provides a queryable, long-retention store for all Intune events.
2
Build a Power BI Dashboard on Graph API
Use the Microsoft.Graph Power BI connector (not the legacy Data Warehouse connector) to pull live device compliance, inventory, and app install data. Schedule hourly refreshes. This gives you a near-real-time operations dashboard.
3
Use Graph API for Automation-Critical Queries
For any script or automation that depends on accurate device state (compliance, last sync, OS version), query the Graph API directly: GET /deviceManagement/managedDevices. Do not rely on portal exports for operational decisions.
4
Set Up Sentinel Alerts
Connect Intune Log Analytics data to Microsoft Sentinel. Create analytic rules to alert on: devices going non-compliant, sudden spike in enrollment failures, or devices not syncing for >24h.
5
Enable Endpoint Analytics for Performance Baseline
Enrol devices into Endpoint Analytics (Intune → Reports → Endpoint Analytics → Enrolment). This provides startup performance baselines and app reliability scores that are not available anywhere else.
✓
Bottom Line
The reporting gap is real but improving. For organisations that need operational-grade monitoring today, the answer is: Graph API + Log Analytics + Power BI. The Intune portal reports are sufficient for day-to-day management, but any compliance-critical or audit-critical reporting workflow should be built on the Graph API or the Log Analytics export.
