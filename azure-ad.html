Architecture
Entra ID + Intune Together

Entra ID is the identity control plane. Intune is the device control plane. Together they form the foundation of Microsoft's Zero Trust access model.

🔐
Entra ID (Azure AD)
Cloud identity provider. Authenticates users and devices. Issues OAuth2/OIDC tokens. Stores device objects including compliance state synced from Intune. Enforces Conditional Access at every token issuance.
📱
Intune Device Management
Manages device configuration and compliance. When a device enrolls, Intune writes the compliance state back to the Entra ID device object. CA reads this state when making access decisions.
Compliance engine
🚦
Conditional Access Engine
Policy engine evaluating: user identity, device compliance, location, app, risk level. Runs at every token issuance. Can grant, block, or require additional verification (MFA, Terms of Use, etc.).
🔄
Continuous Access Evaluation (CAE)
Revokes tokens in near-real-time (not just at expiry). If user is disabled or device becomes non-compliant, access is cut within minutes, not hours.

Device Registration
Three Device Registration Types

Each registration type offers a different level of trust and management in Entra ID.

TYPE 1
Entra ID Registered
Personal/BYOD — user adds work account to personal device
What Happens
Device object created in Entra ID. Work apps get SSO via Workplace Join certificate. Device NOT managed by Intune unless enrolled separately.
CA Trust Level
Registered ✅ | Joined ❌ | Compliant ❌
Use Case
BYOD with app protection only (MAM-WE)
MAM-WE enrollment
TYPE 2
Entra ID Joined
Corporate cloud-only — joined at OS level via Autopilot or OOBE
What Happens
Device joins Entra ID. Gets Primary Refresh Token (PRT) in TPM. MDM auto-enrollment triggered. User logs in with Entra ID credentials only.
CA Trust Level
Registered ✅ | Joined ✅ | Compliant ✅ (if Intune met)
Use Case
Modern cloud-only Windows endpoints
Autopilot provisioning
TYPE 3
Hybrid Entra ID Joined
On-premises AD + Cloud — joined to both domains
What Happens
Joined to on-prem Active Directory. Azure AD Connect syncs device object to Entra ID. Gets both on-prem and cloud identity.
CA Trust Level
Registered ✅ | Hybrid Joined ✅ | Compliant ✅ (if Intune managed)
Use Case
Hybrid environments with legacy AD dependency
Always-On VPN for Hybrid Join
Migration Path
Most orgs move: Hybrid Joined → Entra ID Joined over time via Autopilot refresh cycles. Cloud-only join is the modern target state.
CA Policy Examples
Real-World Conditional Access Configurations

Each CA policy has conditions (IF) and access controls (THEN). These are the most critical policies for any Intune deployment.

🔒 POLICY 1 — Require Compliant Device for M365
IF: All users · Office 365 apps · Any device platform · Any location
THEN: Grant — require device marked compliant
Effect: Non-compliant devices (outdated OS, missing BitLocker, MDE High risk) blocked from all M365 resources.
Note: Exclude break-glass accounts. Add grace period in Intune compliance policy (not CA).
Compliance evaluation engine
🔐 POLICY 2 — Require MFA Outside Trusted Network
IF: All users · All cloud apps · Exclude trusted IP ranges (office networks)
THEN: Grant — require MFA
Effect: Working from office = seamless. Working from home/travel = MFA prompt. Authenticator App push or FIDO2/WHfB recommended (not SMS).
Session: Sign-in frequency 1 hour for unmanaged devices.
🚫 POLICY 3 — Block Legacy Authentication
IF: All users · All cloud apps · Client apps: Exchange ActiveSync, Other clients (IMAP, POP3, SMTP)
THEN: Block access — no exceptions
Effect: Forces modern auth (MSAL) only. Critical — 99% of password spray attacks use legacy auth protocols. Implement this first.
🛡️ POLICY 4 — Phishing-Resistant MFA for Admins
IF: Directory roles: Global Admin, Security Admin, Privileged Role Admin · All cloud apps
THEN: Grant — require authentication strength: Phishing-resistant MFA
Effect: Admins must use FIDO2 security key or Windows Hello for Business. SMS/Authenticator app not accepted for admin accounts.
Authentication Flow
OAuth2 Token Flow — End to End

From opening an app on a managed device to receiving an access token that grants resource access.

🔄 Managed Windows Device → SharePoint (Silent SSO)
1
User opens SharePoint in Edge
Browser detects no session. Redirects to login.microsoftonline.com. MSAL initiates auth flow and checks for an existing Primary Refresh Token (PRT) on the device.
2
PRT Found — Silent SSO
Device has a valid PRT stored in TPM (issued at Entra ID join time). WAM (Web Account Manager) silently exchanges PRT for an access token. No user prompt — seamless SSO.
3
Conditional Access Evaluated
CA policy engine fires. Checks: device compliant? (reads Intune state from Entra ID device object). Sign-in risk low? Location trusted? MFA satisfied by PRT?
4
JWT Access Token Issued
CA passes. Entra ID issues signed JWT with claims: UPN, deviceId, compliance state, MFA method, token lifetime (1 hour). SharePoint validates this token.
5
Resource Access Granted
SharePoint validates the JWT signature using Entra ID's public key. Reads claims to determine permissions. User gets access. Token refreshes every hour silently via PRT.
