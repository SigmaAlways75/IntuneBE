Policy Types
Eight Policy Categories

Each policy type serves a distinct purpose. Choosing the right type determines conflict resolution priority, targeting options, and delivery behaviour.

⚙️
Configuration Profiles
The main policy type. Configure OS settings, Wi-Fi, VPN, email, certificates, OMA-URI. Built on the Settings Catalog or legacy templates. Multiple profiles can apply to one device.
Settings Catalog details
✅
Compliance Policies
Define the minimum security bar. Evaluate device health and set compliance state. Result feeds into Conditional Access. Not enforcing — only reporting state.
Compliance engine
📦
App Protection Policies (MAM)
Applied to apps, not devices. Control copy/paste, save-to-storage, offline grace period. Work without device enrollment. Targeted to users.
MAM SDK deep dive
🛡️
Endpoint Security Policies
Focused security policies: Antivirus, Disk Encryption, Firewall, ASR, EDR, Account Protection. Designed for security teams.
Endpoint security types
🔄
Update Policies / Rings
Control Windows Update for Business, iOS/macOS update schedules. Deploy in rings: Pilot → Broad → Critical. Defer feature updates up to 365 days.
📜
Scripts (PowerShell / Shell)
Run PowerShell on Windows (via IME) or bash/zsh on macOS. One-time or recurring. Runs as SYSTEM or logged-in user. Output logged to Intune.
IME agent details
🏷️
Scope Tags (RBAC)
Role-based access control for Intune admins. Scope Tags limit which admins can see and manage which devices/policies. Essential in large orgs with regional IT teams.
🎯
Assignment Filters
Dynamic targeting at assignment time using device properties (OS version, model, manufacturer). Applied alongside group assignments without creating extra groups.

Conflict Resolution
Policy Priority Order

When multiple policies configure the same setting, Intune resolves the conflict using this priority order — highest wins.

① Endpoint Security Policies — HIGHEST
Security-focused policies (Antivirus, Firewall, ASR) always win over Configuration Profiles for the same setting. Designed so security settings cannot be overridden by general configuration.
② Security Baselines
Pre-built hardening templates. Take precedence over standard Configuration Profiles. Can be overridden by Endpoint Security policies. Conflicts between baselines and profiles are flagged in reporting.
③ Configuration Profiles — User-Targeted
User-assigned profiles apply when user logs in. User-targeted wins for user-scoped settings. Higher than device-targeted for those settings.
④ Configuration Profiles — Device-Targeted
Device-assigned profiles apply regardless of who is logged in. Applied at device check-in. Lower priority than user-targeted for user-scope settings.
⑤ Platform Defaults — LOWEST
OS and Intune defaults apply where no policy is configured. These are the out-of-box settings that all configured policies override.
⚠️ What Happens on a Real Conflict?
Example: Two Configuration Profiles both configure BitLocker encryption method:
→ Profile A (All Devices): AES-128    → Profile B (Finance Team): AES-256

For a Finance device, both apply → CONFLICT detected.
Intune marks the setting as "Error" in the device profile state. Device falls back to OS default or previously applied value.

Resolution: Use Assignment Filters instead of separate groups, or remove the conflicting setting from Profile A.
Targeting
Assignment & Targeting

How policies reach the right devices — group assignment, exclusions, Filters, and the delivery pipeline from Intune to endpoint.

👥
Entra ID Groups
Dynamic or assigned groups. Dynamic groups use rules (e.g., all devices where OS = Windows). Exclusion groups override inclusion — if device is in both, exclusion wins.
🌐
All Devices / All Users
Built-in targets. "All Devices" targets every Intune-enrolled device. "All Users" targets all licensed users. Use for baseline policies that should apply universally.
🔍
Assignment Filters
Refine group assignments at delivery time using device properties. Filter to Windows 11 only, or Surface devices only — without extra groups. Evaluated per-policy at assignment time.
⏱️
Delivery Timing
Policies queue immediately on save. Devices receive on next MDM check-in (default 8hr). WNS/APNs push can trigger immediate check-in. Manual sync available from Company Portal or Settings.
🔍 Assignment Filter Syntax Examples
// Windows 11 devices only
(device.osVersion -startsWith "10.0.22")

// Microsoft Surface devices only
(device.manufacturer -eq "Microsoft") and (device.model -startsWith "Surface")

// iOS 17+ iPhones only
(device.osVersion -startsWith "17.") and (device.deviceType -eq "iPhone")

// Exclude personal devices (Entra Registered only)
(device.deviceOwnership -eq "Company")

API Access
Graph API Policy Operations

Every Intune policy is accessible via Microsoft Graph API — enabling backup/restore, bulk operations, and full Infrastructure as Code workflows.

⚡ Common Graph API Calls
// Base: https://graph.microsoft.com/v1.0/deviceManagement/

// List all configuration policies
GET /configurationPolicies

// Create compliance policy
POST /deviceCompliancePolicies
{
  "@odata.type": "#microsoft.graph.windows10CompliancePolicy",
  "displayName": "Win10 Standard Compliance",
  "osMinimumVersion": "10.0.19041",
  "bitLockerEnabled": true,
  "secureBootEnabled": true
}

// Assign policy to a group
POST /deviceCompliancePolicies/{id}/assign
{ "assignments": [{ "target": { "groupId": "..." } }] }
