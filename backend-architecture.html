Intune Backend Architecture

A complete technical map of how Microsoft Intune's cloud backend communicates with Windows, iOS, Android, macOS, and Linux endpoints — covering every protocol, service, agent, channel, and Azure dependency involved in device management.

01
Architecture
High-Level Backend Architecture

Intune is a cloud-native SaaS service built on Microsoft Azure. Every device management operation flows through a chain of Azure services before reaching the endpoint. Understanding this chain is essential for network planning, firewall rules, and diagnosing connectivity failures.

☁️
Intune Service (Azure)
The central management plane hosted in Microsoft Azure. Exposes the Microsoft Graph API for all management operations. Stores device objects, policies, app assignments, and compliance state.Graph API reference
🔐
Microsoft Entra ID
Identity and device registration backbone. Stores device objects with compliance state. Evaluates Conditional Access policies on every token request. Issues OAuth2/OIDC tokens to devices and users.Entra ID & CA deep dive
📦
Azure Blob Storage
Stores Win32 app packages (.intunewin), PowerShell scripts, and custom content. Devices download content via time-limited SAS URLs directly from Azure CDN — not through the Intune service itself.
🔔
Push Notification Services
Windows Notification Service (WNS) for Windows, Apple Push Notification Service (APNs) for iOS/macOS, Firebase Cloud Messaging (FCM) for Android. Used to wake devices and prompt them to check in for new policies.
📊
Microsoft Defender for Endpoint
Provides real-time device risk signals (risk score: Clear/Low/Medium/High) consumed by Intune compliance evaluation. MDE-detected threats can instantly flip a device to non-compliant.MDE integration
🌐
Azure CDN
Content delivery network for distributing Win32 app packages and scripts globally. Devices download from the closest CDN edge node. Reduces latency for large app deployments across geographically distributed workforces.
📋
Microsoft Graph API
The unified REST API surface for all Intune operations. Every action in the Intune portal — creating a policy, assigning an app, triggering a remote wipe — is a Graph API call under the hood.Key endpoints
ℹ️
Core Architecture Principle
Intune is designed as a push-pull hybrid system. Intune uses push channels (WNS, APNs, FCM) to notify devices that changes are waiting. The device then pulls the actual policy payload from the Intune service over an authenticated HTTPS connection. The device always initiates the data transfer — Intune never pushes raw policy data directly.

All device-to-Intune communication is outbound HTTPS (port 443) from the device. No inbound connections from Intune to devices are required. Windows Deep Dive
Windows MDM Communication Stack

Windows has the richest and most complex Intune integration. The built-in MDM client handles policy delivery, while the Intune Management Extension (IME) handles everything the MDM protocol cannot — Win32 apps, PowerShell scripts, and custom inventory.

Windows MDM Client — OMA-DM Protocol

The Windows MDM client is built directly into the OS (no agent to install). It communicates with Intune using the OMA-DM protocol over HTTPS. Policies are delivered as Configuration Service Provider (CSP) key-value pairs.

The IME is a Windows service that handles capabilities beyond the OMA-DM protocol: Win32 app installation, PowerShell script execution, custom compliance scripts, and proactive remediations.

Windows Enrollment Flow — Technical Sequence

1
User or Autopilot Triggers Enrollment
Enrollment can be triggered manually (Settings → Accounts → Access Work or School), automatically via Autopilot/OOBE, or silently via Group Policy for Hybrid Join. All paths converge at the MDM enrollment endpoint.
2
MDM Discovery
Device calls enterpriseenrollment.{domain}.com (DNS CNAME) or the Intune enrollment endpoint directly to discover the MDM service URL. This CNAME must be configured in your public DNS for user-driven enrollment.
3
Entra ID Device Registration
Device registers with Entra ID. A device certificate is generated and stored in the TPM (for Entra ID Joined devices). The Primary Refresh Token (PRT) is created and bound to the device.
4
MDM Enrollment Certificate Issued
Intune issues a client certificate to the device — the MS-Organization-Access certificate. This cert is used for mutual TLS authentication on all subsequent MDM check-ins. Stored in the Personal certificate store.
5
Initial Policy Sync
Device performs its first check-in. Intune pushes all assigned configuration profiles, compliance policies, app assignments, and certificate profiles. This is the baseline state of the device.
6
IME Auto-Install
If Win32 apps or PowerShell scripts are assigned, the MDM policy triggers installation of the IME from the Microsoft Store. Once installed, IME begins polling independently for its own assignments.
7
Ongoing Check-in Cycle
Every 8 hours (or immediately when WNS push received), the device checks in. Delta sync — only changed or new policies are delivered. Device reports compliance state, inventory, and app install status back to Intune.
🔑
TPM and the MDM Certificate
On modern Windows devices, the MDM enrollment certificate's private key is generated inside the TPM and is hardware-bound — it cannot be exported or copied to another device. This means the MDM trust is tied to the physical hardware, providing a strong device identity guarantee that Intune and Conditional Access rely on. Run certmgr.msc → Personal → Certificates and look for MS-Organization-Access to see the certificate. Troubleshoot MDM cert issues

Apple Platforms
iOS, iPadOS & macOS MDM Stack

Apple platforms use the Apple MDM Protocol — a vendor-specific HTTPS-based protocol using APNs as the push channel. All management is agentless: the MDM profile installed during enrollment is the management channel.

Apple MDM Protocol Flow

1
MDM Profile Installed
During enrollment (ADE or Company Portal), Intune pushes an MDM profile to the device. This profile contains: the MDM server URL, identity certificate, and management capabilities declared by the server.
2
APNs Channel Established
The device registers its APNs device token with Intune. Intune stores this token. All future management commands are initiated by sending a small push notification to this token via the Apple Push Notification Service.
3
APNs Push Triggers Check-in
When Intune wants to send a command (policy, app install, remote wipe), it sends a push notification to the device via APNs. The notification contains no payload — it simply tells the device to check in.
4
Device Polls for Commands
The device connects to the Intune MDM endpoint (*.manage.microsoft.com) over HTTPS using mutual TLS (device identity certificate + Intune server cert). Intune responds with a queued MDM command.
5
Device Executes Command and Reports
Device executes the command (apply profile, install app, etc.) and sends an Acknowledge or Error response back to Intune. Intune removes the command from the queue and logs the result.
6
Inventory Reporting
Device reports full hardware inventory, installed apps, and OS version at enrollment and periodically. Intune stores this in the device record visible in the portal.
⚠️
APNs Certificate — Critical Operational Requirement
Intune requires a valid Apple MDM Push Certificate to manage any Apple device. This certificate expires annually and must be renewed using the same Apple ID that was used to create it. Renewing with a different Apple ID breaks all existing enrolled devices — they must be re-enrolled. Monitor the certificate expiry in: Intune → Devices → iOS/iPadOS enrollment → Apple MDM Push Certificate. Troubleshoot iOS enrollment failures

Android
Android Enterprise MDM Stack

Android Enterprise uses Google's Enterprise Mobility Management (EMM) API, not a proprietary protocol. Intune communicates with Google's cloud, which in turn manages the device via the Android Device Policy app.

Android Enterprise Architecture

🤖
Google EMM API
Intune calls Google's Enterprise API to: push app assignments, apply policies, trigger compliance checks, and manage the device. Intune acts as an EMM provider registered with Google.
📦
Managed Google Play
All Android Enterprise app distribution goes through Managed Google Play. Intune approves apps in the Managed Google Play iframe, then assigns them. Google handles delivery to the device.
🔒
Android Device Policy
Google-issued app that Intune installs in the Work Profile or Fully Managed device. It enforces all Intune policies on the Android side. The device admin agent.
🔔
FCM Push
Firebase Cloud Messaging delivers push notifications to trigger device check-in. Requires fcm.googleapis.com to be reachable. No FCM = no push sync (device still polls every 8h).

ℹ️
Android AOSP (GMS-Free) Devices
For Android devices without Google Mobile Services — common in China or in certain manufacturing hardware — Intune provides a separate management track using the Microsoft Intune app (not Company Portal) and Microsoft's own push channel instead of FCM. This supports a subset of Android Enterprise capabilities.

🚫
SSL Inspection Warning
Do not apply TLS break-and-inspect (SSL inspection) to any of the Intune, Entra ID, APNs, or WNS endpoints. SSL inspection replaces the server's certificate with the proxy's certificate. MDM protocols and certificate pinning used by Apple APNs, WNS, and the Intune MDM client will fail if the certificate chain is altered by an intermediate proxy. Add all domains listed above to your proxy/firewall SSL inspection bypass list.

MAM Architecture
App Protection Without Device Enrollment

The MAM-WE (Mobile Application Management Without Enrollment) path manages data inside apps without any device-level management. This is architecturally separate from the MDM path and uses a different authentication and policy delivery mechanism.

MAM Architecture — How It Differs from MDM

🔑
MSAL Token-Based
No MDM profile. No device certificate. Instead, the Intune SDK inside the app uses the user's MSAL access token to authenticate to the Intune MAM service and retrieve the App Protection Policy assigned to that user.
📦
App-Layer Enforcement
All policy controls (copy/paste block, save restrictions, PIN requirement) are enforced inside the app by the SDK. The OS is not involved — no MDM profile, no CSP settings.
🔄
Policy Refresh Cycle
The SDK refreshes the App Protection Policy every 30 minutes while the app is in the foreground, and on every app launch if more than 30 minutes have elapsed since the last refresh.
🧹
Selective Wipe Path
IT issues a selective wipe via the Intune MAM service. On the next token refresh, the SDK detects the wipe flag and removes all corporate data from the app's managed storage. Device is completely untouched.

Policy Lifecycle
Policy Delivery & Device Check-in Lifecycle

How policies move from an admin saving them in Intune to the setting being applied on the device — and what happens during every subsequent check-in.

1
Admin Saves a Policy in Intune
Policy is written to the Intune database. The change is immediately queued for delivery. Intune sends a push notification (WNS/APNs/FCM) to all devices targeted by the policy's assigned group.
2
Push Notification Triggers Check-in
The device's push notification client receives the signal and wakes the MDM client. The MDM client opens an HTTPS session to the Intune MDM endpoint using its enrollment certificate for mutual TLS authentication.
3
Delta Sync — Changed Policies Only
Intune performs a delta sync — only policies that have changed since the last check-in are sent. This keeps check-in traffic minimal even on devices with hundreds of assigned policies.
4
Device Applies Policy Settings
The MDM client writes the received CSP values to the appropriate registry locations or applies the configuration profile. For Win32 apps and scripts, the IME processes its assignments separately in its own polling cycle.
5
Device Reports Compliance State
After applying policies, the device evaluates all compliance policies against its current state and reports the result back to Intune. Intune updates the device's compliance state in Entra ID.
6
Conditional Access Reads Compliance
On the user's next token request (accessing Outlook, Teams, SharePoint), Conditional Access reads the compliance state from the Entra ID device object and makes an access decision.
7
Scheduled Check-in Every 8 Hours
Even without a push trigger, the device checks in on its own schedule (default every 8 hours for all platforms). This ensures policies are never more than 8 hours stale even if push notifications are blocked.
⚡
Forcing an Immediate Check-in
You can trigger an out-of-cycle check-in three ways:
1. From Intune portal: Devices → [Device] → Sync
2. From Windows device: Settings → Accounts → Access Work or School → Info → Sync
3. Via Graph API: POST /deviceManagement/managedDevices/{id}/syncDevice  Graph API remote actions
ℹ️
Check-in Frequency Reference
After initial enrollment: 1 min, 3 min, 5 min, 15 min, then hourly for 8 hours, then every 8 hours. The aggressive initial schedule ensures newly enrolled devices receive all their policies quickly. After the device stabilises, it settles into the 8-hour cadence.
