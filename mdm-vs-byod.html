MDM vs BYOD at a Glance

MDM (Corporate-Owned) gives IT full device control. BYOD uses app-layer or Work Profile isolation to protect corporate data while respecting user privacy.

Corporate-Owned
Full MDM Control
IT manages the entire device — OS, apps, settings, hardware
Control Level
Full device — wipe, lock, configure OS, push any policy
Enrollment
ADE/DEP (Apple), Zero-Touch (Android), Autopilot (Windows)
Remote Wipe
Full factory reset — all data including personal erased
Security Posture
Strongest — BitLocker/FileVault enforced, cert-based auth
IT Visibility
Full hardware inventory, installed apps, location (if enabled)
User Privacy
Minimal — device is company property
Employee-Owned
BYOD / MAM
IT manages only work apps and data — personal content untouched
Control Level
App-level only (MAM) or Work Profile (Android) — not full device
Enrollment
Company Portal app, or MAM-WE (no enrollment needed)
Remote Wipe
Selective wipe — only corporate data removed, personal stays
Security Posture
Good — PIN/biometric on work apps, encryption in container
IT Visibility
Limited — only work app data and compliance state visible
User Privacy
High — personal apps and data never visible to IT
Four Enrollment Models
🏢
COPE — Corporate Owned, Personally Enabled
Company owns device but allows personal use. IT has full control but creates a Work Profile for separation. Best balance of security and user experience.
Zero Trust for COPE
🔐
COBO — Corporate Owned, Business Only
Fully locked down. No personal apps. Used for kiosk devices, frontline workers. Managed via Dedicated Device enrollment on Android.
Policy assignment for COBO
📱
BYOD — Bring Your Own Device
Employee device, IT manages only work apps via MAM or Android Work Profile. Personal data completely isolated. User can unenroll anytime.
App Protection Policies
☁️
MAM-WE — Without Enrollment
No device registration. Intune app protection policies applied directly to Office apps via MSAL token. Zero device footprint for IT.
CA policy for MAM-WE

Technical Flows
Enrollment Flows by Platform

Step-by-step technical flows for each major enrollment scenario.

🍎 Apple ADE — Automated Device Enrollment (Corporate)
1
Purchase → Apple Business Manager
Devices purchased via Apple or authorised reseller are automatically linked to the org's ABM account by serial number. No physical IT touch required.
2
Intune Syncs ABM
Intune pulls the device list from ABM via Apple REST API. IT assigns an ADE enrollment profile specifying MDM server URL, supervised mode, and which OOBE screens to skip.
3
First Boot → Auto-Enroll
Device contacts albert.apple.com on power-on. Apple returns the MDM enrollment profile. Device contacts Intune MDM endpoint and sends its identity token.
4
MDM Check-In & Policy Push
Intune pushes all configuration profiles, certificates, and required apps silently via APNs. Device fully managed before user logs in.
Windows equivalent: Autopilot
🤖 Android BYOD — Work Profile Enrollment
1
User Installs Company Portal
Employee downloads the Intune Company Portal from Google Play and signs in with their work Microsoft 365 account.
2
Work Profile Created
Company Portal triggers Android OS to create a cryptographically isolated Work Profile container. Personal and work apps separated at kernel level.
3
Intune ↔ Google EMM API
Intune registers the device with Google's Enterprise API. Android Device Policy app pushed into the Work Profile. Policy tokens exchanged.
4
Work Apps Deployed
Managed apps appear in Work Profile with badge icon. IT can push, remove, and configure apps inside the container. Personal side untouched.
☁️ MAM Without Enrollment — iOS & Android
1
User Opens Outlook / Teams
No Company Portal needed. MSAL library in the app initiates Azure AD authentication as soon as the user signs in with their work account.
2
Intune SDK Checks APP
During token issuance, Intune checks if the user has an App Protection Policy assigned. The Intune SDK embedded in the app contacts the Intune service.
3
Policy Enforced In-App
Without any device enrollment, copy/paste restriction, PIN requirement, screen capture block, and save-to-personal-storage block are all enforced inside the app sandbox.
4
Selective Wipe on Departure
IT issues a selective wipe via Intune. On next token refresh the SDK detects the wipe flag and removes all corporate data from the app. Device untouched.

Data Isolation
Privacy & What IT Can See

Intune BYOD is designed to respect user privacy — IT sees only what is necessary for security. Here is exactly where the boundary sits.

👁️
What IT CAN See (BYOD)
Device model and OS version
Device compliance state (yes/no)
Work app inventory (in Work Profile)
Encryption status
Jailbreak/root detection result
Last sync time
🚫
What IT CANNOT See (BYOD)
Personal app list
Personal photos, messages, calls
Browser history
Location (not enabled by default)
Personal email or contacts
Any data outside Work Profile
🔐
Android Work Profile Isolation
Hardware-enforced cryptographic container. Apps in personal and work space cannot share data via Intents, clipboard, or file system. Separate encryption keys at the Android keystore level.
📦
iOS App Sandbox Isolation
Intune App Protection Policies block managed-to-unmanaged data transfer. Copy from Outlook → Personal Notes is blocked. Save to iCloud from managed apps is blocked. Enforced by Intune SDK inside the app.
Guidance
Which Model Should You Choose?

Use case-based decision guide for selecting the right enrollment model.

🏥 High Security / Regulated (Healthcare, Finance, Gov)
→ COBO or COPE with full MDM
→ Enforce BitLocker/FileVault, certificate-based auth
→ Enable Conditional Access requiring compliant device
→ Deploy via ADE (iOS/macOS) or Windows Autopilot
→ Enable Microsoft Defender for Endpoint integration
Endpoint Security & Compliance
🏢 Mid-Size Enterprise — Mixed Workforce
→ Corporate laptops: Full MDM via Autopilot/ADE
→ Employee phones: BYOD with Android Work Profile
→ Remote contractors: MAM-WE (no enrollment)
→ Conditional Access: require MFA + compliant or app protection
Windows Autopilot for laptops
🚀 Startup / SMB — Minimal IT Overhead
→ Start with MAM-WE (no device enrollment)
→ Configure App Protection Policies for Outlook, Teams, SharePoint
→ Enable MFA for all users via Conditional Access
→ Selective wipe on departure handles offboarding
→ Upgrade to full MDM as you scale
Configure Conditional Access
🏭 Frontline / Deskless Workers (Retail, Manufacturing)
→ Use COBO with Android Dedicated Device (kiosk mode)
→ Single-app or multi-app kiosk via Managed Home Screen
→ Zero-Touch Enrollment for bulk device provisioning
→ Shared device mode — multiple workers, one device
