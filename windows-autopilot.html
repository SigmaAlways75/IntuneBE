What is Autopilot?
Zero-Touch Windows Provisioning

Autopilot is a cloud-native zero-touch provisioning system. Devices ship directly to users — on first power-on they automatically enroll into Intune, apply all policies and apps, and are fully managed before the user reaches the desktop. No imaging. No IT hands-on.

🏭
Hardware Hash Registration
Each device is identified by a 4K hardware hash. OEMs or IT register this hash to your tenant via Partner Center or PowerShell. Once registered, the device "knows" to join your tenant on first boot.
📋
Deployment Profiles
Profiles define the OOBE experience: screens to skip (EULA, privacy, Cortana), join type (Entra ID vs Hybrid), naming template, deployment mode. Assigned to device groups in Intune.
Profile assignment & targeting
📊
Enrollment Status Page (ESP)
Progress screen shown during provisioning. Tracks 3 phases: Device Preparation, Device Setup (policies + certs + apps), Account Setup (user apps). IT can block desktop access until complete.
🔐
Entra ID Join at OOBE
Device joins Entra ID and triggers Intune MDM auto-enrollment simultaneously. PRT issued and stored in TPM. All subsequent app sign-ins use SSO — no passwords needed.
Entra ID join types
🔑
Certificate Delivery at Enrollment
SCEP certificate profiles deploy during the Device Setup phase of ESP. Wi-Fi and VPN certificates delivered before the user logs in — ensuring seamless corporate access from first sign-in.
SCEP certificate flow
🧹
Autopilot Reset
Re-provisions an existing device without re-imaging. Wipes all user data and apps, returns to ESP state, re-applies all Intune policies. Used for device refresh between employees.
Deployment Modes
Four Ways to Deploy via Autopilot

Each mode suits different scenarios — from remote workers to kiosk devices to enterprise fleet migration.

👤 1. User-Driven Mode — Most Common
User receives device, powers on, connects to Wi-Fi, signs in with work account. Autopilot profile loads automatically. ESP runs in background installing policies and apps. User reaches desktop once ESP completes.

Join types: Entra ID Joined (cloud-only) or Hybrid Entra ID Joined (requires on-prem connector).
Best for: Remote workers, distributed deployments, devices shipped directly to users.
🤖 2. Self-Deploying Mode — Zero User Interaction
No user credentials required. Device provisions fully automatically using TPM 2.0 device attestation. Device authenticates to Entra ID using the TPM certificate — not user credentials.

Requirement: TPM 2.0 mandatory.
Best for: Frontline kiosks, shared workstations, meeting room devices, digital signage.
COBO enrollment model
🧤 3. Pre-Provisioning (White Glove) — IT Pre-stages Device
IT or reseller runs the "Technician flow" before the device reaches the user. Device Setup phase (policies, certs, Win32 apps) completes fully. Device resealed and shipped. User only completes the fast Account Setup phase.

Result: User gets a fully provisioned device in minutes instead of 30–60 min for full ESP.
Best for: Large deployments, slow WAN links at branch offices.
🔄 4. Autopilot for Existing Devices — Re-purposing Older Hardware
Used with SCCM (ConfigMgr) to re-deploy existing on-prem managed devices as cloud-managed Autopilot devices. A task sequence wipes the device, registers the hash, and triggers Autopilot enrollment on reboot.

Best for: Migration from SCCM to full cloud management, fleet modernisation.
Hybrid Entra ID Join
Technical Flow
User-Driven Autopilot — Step by Step

The complete sequence from device power-on to fully managed desktop.

🔄 Complete Provisioning Flow
1
Power On → OOBE Starts
Windows OOBE launches. Device connects to network. Windows contacts ztd.dds.microsoft.com (Autopilot service) with the device's hardware hash to check for an Autopilot profile.
2
Autopilot Profile Downloaded
Autopilot service matches the hash to the tenant. Returns the deployment profile — skip EULA: yes, skip privacy: yes, user-driven mode, naming template: CORP-%RAND:5%. OOBE customises itself.
3
User Signs In with Work Account
OOBE shows Microsoft sign-in page. User enters Entra ID UPN and password. MFA may be required depending on Conditional Access policy.
4
Entra ID Join
Device joins Entra ID. Device certificate issued and stored in TPM. PRT created. Device object created in Entra ID with deviceId, join type, and registrationTime.
5
MDM Auto-Enrollment
Entra ID sees MDM auto-enrollment configured for this user/group. Triggers Intune enrollment automatically. Device POSTs to Intune DM enrollment endpoint with the Entra ID token.
6
ESP — Device Setup Phase
ESP appears. Device policies download and apply (BitLocker, firewall, compliance). SCEP certificates deploy. Required Win32 apps start installing via IME. Longest phase — 10–45 min.
7
ESP — Account Setup Phase
User-targeted policies apply. Office 365, Teams install. OneDrive Known Folder Move silently configures. User-scoped certificates deploy.
8
Desktop — Fully Managed
User reaches desktop. All policies applied, apps installed, certificates in place. BitLocker active. Device is Compliant in Intune. CA grants full M365 access seamlessly.
ESP Phases
Enrollment Status Page — Deep Dive

ESP ensures a device is fully configured before the user accesses the desktop. Understanding its phases is key to diagnosing slow or stuck deployments.

⚙️
Phase 1 — Device Preparation
Device registers with Intune. MDM certificate issued. Device inventory sent. Typically under 1 minute. If this hangs, check network connectivity to Intune endpoints.
📥
Phase 2 — Device Setup
Device-targeted profiles apply. BitLocker, firewall, certificates, Wi-Fi/VPN. Required Win32 apps install via IME. Longest phase — 10–45 min depending on app count/size.
SCEP delivery in ESP
👤
Phase 3 — Account Setup
User-targeted policies and apps apply after sign-in. Office 365, Teams install. OneDrive silently configures. Faster than Phase 2 since heavy apps already installed.
⏱️
Timeout & Blocking
Configure: block desktop until ESP completes · set timeout (15–720 min) · allow user bypass on timeout (safety net). Recommended: 60 min timeout with bypass allowed.
🔍
Diagnostic Logs
If ESP fails: press Ctrl+Shift+D for diagnostics panel. Check MDMDiagReport.html. Event logs: DeviceManagement-Enterprise-Diagnostics-Provider.
📦
Required vs Available Apps
Only Required apps block ESP. Available apps do not. Minimise Required apps to keep ESP fast. Office 365 and security tools are typical Required candidates.
